Motivation

Poof was born out of frustration with the lack of quality of life features in C and C++. Where C lacks features entirely, C++ has ergonomics ranging from tolerable to outrageously poor. I wanted a quick an easy way to express simple transformations, primarily using strings, and sometimes querying the structure of the program itself. Thus, poof was born.

Goals

Poof stands upon three main tenants:

Be simple

A statement that's much easier said than done. With the same spirit as C, I've tried as best I can to limit the scope of the language to simple features that allow a user to express what they want, how they want to. At present, implementations of most poof features fit on the back of an index card.

Be unintrusive

poof strives to be there for the programmer when they need it, and then GTFO. In concrete terms this means the compiler is fast. The code it generates is readable, debuggable, and mirrors the style of the project. If you want to stop using it at any point, it's (almost) like you never used it at all.

poof does not assume anything about the structure of your project. You tell it when, where, and how to generate code.

Be useful

poof does not ascribe to any dogmatic ideas of how a language 'should be'. Poof language features are born out of pure pragmatism. If it's in the language, it's useful for solving a real, concrete problem that users actually have.

Ever wanted to quickly print an enum (or struct) value to the console?

Serialize that thing to disk?

Poke around in that hashtable at runtime to see what's in it?

Enter poof.
